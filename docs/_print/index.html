BFF技术文档

1 - 概览

Backends For Frontends 服务于前端的后端

BFF

Backends For Frontends 服务于前端的后端

简介

鉴于虚拟化平台内部包含多个Web服务，包括：运营管理平台、云安全控制台、虚拟化平台管理界面等，这些服务有一部分接口是重复的，并且像认证、权限、安全、和前端对接等方面的处理逻辑基本是相同的。基于现在CBB的开发思想，我们考虑能不能有个框架实现管理平台的共有功能，避免大家做太多重复工作，同时提高产品开发效率。

这里设计了BFF（服务于前端的后端），实现和前端框架对接和基础的公共模块封装。

BFF不同于常说的web框架，比如echo、go-zero，它是基于web框架开发的业务框架。

BFF功能

与前端对接

登陆认证、消息中心、前后端数据加密对接、重放攻击等安全组件对接。

内置公共API服务

认证管理、用户管理、系统配置、审计日志、消息中心、下载中心、级联管理、升级管理(文件上传)、大屏展示、CBB组合、k3s集群管理

为内部API服务提供代理

通过rest proxy中间件为产品侧API提供代理。（实现原理同资源池网元二级代理）

BFF框架

就像BFF名称那样，服务于前端的后端，它是和前端框架深度对接的。使用BFF和与BFF配套的前端框架，基本的界面架构和功能就有了。

BFF有三种使用模式：

  • 直连：后端API服务与BFF共享配置，认证、鉴权等都自己做。该模式虽然需要后端API服务自己做一些事情，但是很简单，只需要仿照BFF加载插件就行。
  • 代理：请求先过BFF，认证、鉴权等BFF做，之后代理到后端API服务。该模式下，BFF也可以啥都不做，只做代理，或者只提供前端web服务，或者只加载部分功能。
  • 组合：根据需求，对多个CBB API进行封装，提供更接近产品需求的接口。该模式相当于BFF内部API。

直连模式

在使用直连模式时，BFF作为网关和产品侧API之间的媒介存在。产品侧API服务启动时，从BFF GRPC服务处拉取配置，根据BFF返回的配置启动rest服务，这样保证所有API服务的配置是相同的。配置相同，又基于相同的框架，所有API服务的加密、认证、权限等插件都是互认的，对GW另一侧来说他们是一个服务。

代理模式

所有请求先到BFF，然后经过BFF中的一些列安全插件，最后如果是BFF内部接口，则直接调用对应的handler，如果是其它API服务的接口，则通过代理转给内部的API服务。

上图中只列举了几个关键的rest middlewares。

产品侧API启动时调用BFF RPC拿配置，然后启动服务，启动后再次调用BFF RPC上报自己包含的routes。BFF代理功能是基于rest框架中的proxy插件实现的，该插件基于各产品侧服务上报的routes信息提供代理服务。这里产品侧服务启动时不需要像初期那样，加载太多的中间件，只需要加载recover、logger等最基本的中间件即可。

BFF框架对内部API服务使用的框架没有硬性要求，可以采用rest、go-zero、bahamut等，只要启动时按格式把路由列表提交给BFF就行。

另外BFF还提供前端静态文件服务。这样做有两个特殊的场景：

  • 前后端可以编译为一个可执行文件
  • 前后端默认加密key（在前端js源码里面），可以通过代码动态修改。

如果CBB需要有自己的界面，可以使用这种模式。基本上自己填俩界面就可以了。

组合模式

组合模式用于根据需求整合多个CBB资源，提供更接近产品需求的接口。

1.1 - 在资源池运营管理平台中的应用

运营和控制台

在BFF的设计中，BFF是和管理系统一对一的。一个BFF只能服务于一个管理系统，因为BFF中有用户管理、系统配置、权限认证，当一个BFF服务于多个管理系统时数据会混在一起，导致系统不可用。

在运营管理平台中有个比较特殊的情况，它有控制台。运营9443，控制台443，它们两个是独立的界面。并且控制台是为租户提供的，它的认证系统权限等和运营管理平台不一样。根据我们以往的开发经验（v1开发经验），我们把运营和控制台看成一个系统，当我们提到运营时，一般默认附带着包含控制台。而且v1版运营和控制台所有数据都是共享的，数据库是共享的，甚至前端的配置文件都挂载的同一个目录。

这在使用BFF的时候会遇到一个小问题，BFF的思想是抽离、独立、公用。在实际使用中会涉及到数据交叉问题，控制台怎么和BFF结合，控制台的租户的token怎么生成，BFF怎么进行租户认证，BFF里面包含系统配置那怎么在运营管理平台同时修改运营和控制台的配置。

BFF和运营

如上图，在实际运行的时候存在两个BFF，一个是运营BFF，一个是控制台BFF。这解决了一个BFF不能服务于多个管理系统的问题。并且这两个BFF使用不同的数据库，数据不会混杂。

运营和控制台通过内部RPC接口与BFF进行交互。RPC接口功能包括：用户管理、系统配置、消息推送等。

打包

上图中存在两个BFF，运营BFF和控制台BFF。两个BFF通过挂载不同的配置文件进行区分，这不能避免因人为失误配置文件挂反了导致服务异常的问题。

为此，运营控制台使用BFF时应定制BFF镜像。参考：基于BFF镜像定制产品侧BFF镜像-定制BFF

控制台租户认证

为了统一使用BFF进行登陆认证，用户配置。在控制台BFF使用场景中，需要将租户信息导入控制台BFF。

控制台的租户来源分几类：

  • 在运营租户管理添加租户： 这种运营后台API调用控制台BFF RPC，将租户信息同步给控制台BFF。
  • 在控制台界面添加子用户： 这种控制台后台API调用控制台BFF RPC，将子用户信息同步给控制台BFF。
  • 在运营租户管理导入租户：这种运营后台API调用控制台BFF RPC，将租户信息同步给控制台BFF。

在将租户信息导入控制台BFF后，所有BFF功能就可用了。

租户

我们说租户登陆控制台，租户怎么使用BFF登陆，租户怎么怎么样，其实没有这问题。

首先租户这个概念，只有从研发角度，产品角度，或者运营管理平台管理员角度，才有租户这个东西。如果从安全服务最终使用者的角度，没有租户这个概念。比如我登陆控制台访问我的安全服务，我不认为我是租户，孙子才是租户，我是用户。

同理，站在BFF的角度来说，BFF管理的就是用户，来我这儿登陆的就是我的用户，我只对我的用户负责。访问运营BFF的是运营的用户，访问控制台BFF的是控制台的用户。没有什么租户，租户是什么我不知道。如果要使用，必须把用户信息同步给我。至于同步过来的信息之前是什么玩意儿，和我没关系，我不关心。

运营系统配置

BFF里面包含一些基本的系统配置，比如：系统信息、验证码等，并且这些都有对应的API接口。但是运营管理平台有个比较特殊的地方，有几个地方可以同时修改运营和控制台的配置。这种情况可以这样：

运营自己实现接口。Logic里面运营的配置调用运营BFF RPC，控制台的配置调用控制台BFF RPC。这样配置才会生效。

这种情况把BFF看成数据库吧，看成通过RPC存库就行。

2 - 使用说明

使用BFF的方法和准备工作

2.1 - 基于BFF镜像定制产品侧BFF镜像

基于BFF镜像定制产品侧BFF镜像

BFF是通用的，它里面包含一份基础的配置文件。当产品侧使用BFF的时候，可以基于BFF基础镜像，编译属于产品的特有的BFF。

BFF镜像

BFF镜像是: docker-hub.cloud.top/cbb/bff:dev

它的工作目录是 /app ，该目录下包含执行文件bff和配置文件目录etc。

etc目录下目录结构为:

/app $: tree etc
etc
├── config-map
│   └── permission           # 权限配置文件目录
│       ├── menu.json        #前端菜单配置文件
│       └── rbac_model.conf  # casbin规则
├── i18n                     # 翻译文件
│   ├── en-US.all.json
├── rest-api.yaml            # 配置文件
├── ssl                      # https证书
│   ├── rest-api.crt
│   └── rest-api.key
└── waf                      # waf规则文件

直接使用

如上，产品侧可以直接使用BFF镜像，只需要在运行是挂载一下必要的配置文件, 一般需要挂载：rest-api.yaml menu.json ssl

这种情况下如果产品内部使用了多个BFF，配置文件挂在混了可能导致意外事情发生。

定制BFF

产品侧可以基于BFF镜像，定制属于产品自己的BFF。定制BFF只需修改里面的默认配置文件，使产品侧BFF镜像默认包含产品的配置。

Dockerfile

产品侧定制BFF时Dockerfile参考下面：

FROM docker-hub.cloud.top/cbb/bff:dev

WORKDIR /app
COPY etc /app/etc
EXPOSE 443 8080

CMD ["./bff", "-f", "/app/etc/rest-api.yaml"]

如上，主要是拷贝了etc下面的配置文件。etc下面的目录结构按上节中的创建，COPY时如果BFF已经存在对应的文件，则会覆盖原文件。

定制rest-api.yaml

下面主要列出了需要修改的字段：

# bff rpc服务名称
# 建议必须改

rpc_name: soc.rpc 

# 服务端口

定制权限配置文件

权限配置文件在etc/config-map/permission下面，由于不同产品，菜单不一样，每个菜单调用的url也不一样，所以这下面两个文件应该是产品侧提供的。

定制BFF时，可以默认编译到定制BFF镜像里面，避免每次挂载。

定制ssl文件

各产品应该使用自己的https证书文件，不要使用默认的。

前端文件

默认BFF可以提供前端静态资源访问服务，但里面没有前端文件，需要运行时挂载。在定制BFF时，可以将前端资源文件编译进BFF。

需要将前端文件放到/srv/www/dist目录下。

定制初始化数据

每个产品的初始化数据都是不一样的，可以参考BFF中的初始化脚本写产品自身的初始化数据方法： https://git.cloud.top/cloud/BFF/-/tree/master/rest/shells/init

3.1 - 操作日志

当使用BFF时，操作日志统一由BFF记录，并由BFF提供操作日志查询接口。

BFF记录日志和API服务自己记录日志的实现方式是一样的，只不过由BFF记录，可以保证统一性，并且也比较好实现。

由于BFF记录日志，所有内部API服务中关于日志的i18n翻译文件，需要给BFF一份。可以在部署的时候通过挂载的方式将翻译文件挂载进去。或者API服务启动的时候，调用BFF RPC接口，将i18n文件内容发送给BFF。

整体实现思路，和以前运营差不多。主要是几个日志中的字段意义需要明确一下。

操作日志插件使用说明

  • 目前操作日志支持三种存储方式，分别是clickhouse, mongo 和 文件存储, 需要在配置文件中配置，如下
  # 打印请求日志
  logger:
    enable: true
    type: clickhouse
  • 其中type代表存储方式，可选值为：clickhouse, mongo, file
  • 三种存储方式对外表现完全一致，使用者不需要考虑针对某一种存储方式做特定操作
  • 如果在logic 里需要使用操作日志resource,比如读取（一般情况不需要），应该如下:
l.svcCtx.ApiLog.Default.Read(conditions)

只需要使用Default即可

添加日志详情的方法1

不管是BFF还是其它api服务，方法是一样的：

首先更新go-zero v1.4.3+ goctl tools/goctl/v1.4.1-20220908

logic里面

因为一般逻辑处理在logic里面，go-zero v1.4.3增加了在logic里面记录日志详情的方法:

l.syslog(id, data)

handler里面

在handler里面可以使用

"git.cloud.top/go/go-zero/rest/echo/logger"

logger.SetSyslog(c, syslog)

上面两个地方添加的日志效果一样，选一个地方填加就行。

api服务调用上面方法自定义日志详情后，框架会把这两个值传给BFF，并放到BFF的rest.Context中。所以两边使用起来都和操作本地数据一样。

在使用BFF的情况下，类似/from/:name这种路由，由于是在产品侧api中注册的，BFF中没有路由信息，所以BFF获取的Path是/from/具体值，导致BFF翻译出问题。go-zero v1.4.5修改了该问题。（注意handler中要有logger.BindSyslog(c, l.Context())）

操作日志部分字段说明

username

用户名。注：查询的时候不要返回匿名用户的日志。

另外日志中如果有过多的匿名用户的日志，需要思考日志记得对不对。

menu

前端菜单，表示这个请求是由哪个菜单发起的。

可以对应日志界面的操作对象列, 翻译由前端负责。

level

日志级别。

  • 信息： 20x
  • 警告: 600
  • 错误: 40x
  • 严重错误: 50x

这里有个需求：造某个级别的日志，测评的时候可能会遇到。需要整理一下访问哪个页面，做什么操作会触发对应的日志。

success

是否成功。20x成功，否则失败。

time

本地时间，时间戳。

content

这个字段可以表示日志内容或详情，前端只需要显示就行。日志内容不够详细，后端负责优化。

后端日志content计算方法

if syslogid 非空:
    logid = syslogid
else:
    logid= method+path

content = i18n(logid, detail)

当一个api接口通过method + path翻译不能满足需求时，可以自定义翻译id syslog_id。

如上，syslog_id非空时，日志的id为syslog_id，否则为默认的method + path。

content为日志id经i18n翻译后的内容。

这里的detail是日志中翻译模板中可能用到的值。

使用syslog_id 的情况例子:

// i18n翻译文件
[
  {
          "id": "helloName",
          "translation":"你好 {{.Name}}"
  }
]


// handler里面写日志详情
logger.SetSysLogID(c, "helloName")
logger.Detail(c, '{"Name": "zhangsan"}')

// 翻译
T(ctx, syslog_id, detail)

3.2 - 权限管理

鉴权流程

鉴权流程解释

  • 请求进来先通过jwt中间件，过了之后可以从context中拿到用户名, 进而从如下两方面进行了判断:
  • 一个是判断该用户拥有的menu_id是否和传过来的menu_link相吻合。这个是通过req header中的menu_link来判断。
  • 第二个判断该用户是否有访问某一条路由的权限。

menu(菜单)是什么，从哪里来

  • 菜单和menu ID, 由前端提供一个menu.json文件，文件示例如下:
[
  {
    "text": "主导航",
    "i18n": "menu.main",
    "group": true,
    "hideInBreadcrumb": true,
    "children": [
      {
        "text": "tenant_manage",
        "i18n": "menu.tenant_manage",
        "icon": "anticon antanticon anticon-team",
        "readOnly": false,
        "disabled": false,
        "id": 1,
        "children": [
          {
            "text": "tenant_manage",
            "link": "/user-center/tenant-manage",
            "i18n": "menu.tenant_manage",
            "readOnly": false,
            "disabled": false,
            "id": 11
          }
        ]
      }
    ]
  }
]
  • 文件说明: 这个文件是全部前端可以展示的菜单，菜单是一个树状结构。
  • 字段说明：
  • id：menu id, 此id 唯一；
  • link： menu_link，请求的时候需要放在req header中。
  • disable：此菜单节点是否可用
  • readonly: 此菜单节点是否可读写还是只读

用户、角色、和权限(或者说菜单)的对应关系

  • 用户表里有一个role_name字段对应角色表里的role
  • role表里有一个permissions的字段，是一个数组，对应多个permission, 如下:
{ 
    "role_name" : "dfwer", 
    "role_type" : "custom", 
    "role_desc" : "", 
    "mode" : "", 
    "client_id" : "", 
    "permissions" : [
        "10", 
        "110", 
        "120", 
        "130", 
        "140"
    ]
}
  • permissions 里的元素是 menu id + “0” 或者 menu id + “1”, 0代表只读，1代表可读写

怎么判断用户是否具有某一条路由的权限

  • 从上下文里拿到这条请求的路径，和请求方法(GET PUT DELETE POST)，用户名，通过用户名查到该用户的角色, 然后去查该角色是否有这条路由的权限。
  • 目前是借由casbin实现的，从目前的复杂度来看，其实不用casbin也可以，casbin最适用的场景是角色树，比如租户有子租户的全部权限，子租户还有子租户，疯狂嵌套类似这种。

当开发了新功能模块时，开发人员需要做哪些工作

  • menu.json, 只需要维护这个文件。

10月11日更新

优化使用者 更新权限时，流程过于复杂的问题

说明
  • 目前使用者更新权限，需要更新menu.json, permission.json, bff_user_role和 bff_user_role_t 四个文件，并且执行初始化程序，才能够正常更新权限模块。
  • 改版后，使用者只需要维护menu.json 和permission.json两个文件，并且不用执行初始化程序
  • 新增的这个功能也是函数式的，以后可以根据需要来决定是否使用(目前直接使用)
menu.json变动
  • menu.json增加字段owner , 如下:
{
     "id": "2",
     "text": "订单管理",
     "i18n": "menu.order_management",
     "icon": "anticon antanticon anticon-file-done",
     "link": "/customer/order",
     "owner": ["role_security"]
}
  • owner为拥有该菜单(权限)的角色，这个角色可以是预置角色，也可以不是（一般情况更新预置角色即可），只要是当前数据库里角色表里存在的角色都会更新
  • 超级管理员不用写，自动拥有全部权限，超管的角色名是role_admin 这个是固定的。
  • 如果menu.json没有出现的角色不会作任何更改，出现了的角色，会全量更新，即只要是出现了的角色，menu.json里必须保证此角色在所有应该存在的menu节点下存在。
  • 权限的更新时机是启动服务和切换三权超管时，不用再执行初始化程序。

10月17日更新

删除permission.json文件，合并至menu.json

说明

  • 两个配置文件合并，现在一个完整的menu.json中的节点示例如下:
{
        "id": "1",
        "text": "测试菜单",
        "i18n": "menu.equipment_monitor",
        "icon": "anticon antanticon anticon-compass",
        "owner": ["role_security"],
        "readOnly": false,
        "disabled": false,
        "hide": false,
        "paths": ["/api/v2/monitor/overview*","/api/v2/monitor/device_instance*"],
        "skipper": ["/api/v2/monitor/overview/test"],
        "children": []
}

字段说明

  • id: menu id，必须有，如果没有,此节点包括其子节点都不会有任何作用, 类型是字符串，可以以0开头
  • i18n: 节点名称, icon: 图标, text: 中文名称, hide: 是否隐藏, disable: 是否作废. 这几个是给前端显示用的.
  • readOnly: 只读还是可读写
  • owner: 权限拥有者，见上文
  • paths: 这个菜单下面的所有可访问路径，代替之前的permission.json
  • skipper: 这个菜单下面的所有可跳过路径，代替之前的permission.json(实际上不是这个菜单下的也可)
  • children: 子节点，这是一棵树，可以有无穷多级
凡是不在此列出的字段都是无效的
删除此文档所有和permission.json 相关的内容，以后只维护一个menu.json

3.3 - 消息中心

流程简述

  • web前端通过websocket连接服务端，服务端维持这个连接，直到客户端断开，或者认证过期。
  • 建立连接后，服务端会首先下发一次消息中心统计数据，之后在每次消息中心数据有更新后，都会下发所有客户端一次统计数据。
  • 消息生产者可以通过http和rpc发送消息的api，定向发给某一个或某一组客户端消息
  • 发送消息必须通过消息模版, 消息模版下面有说明。

其他服务和消息中心的交互

  1. 客户端, 客户端通过/ws 接口建立websocket连接
  2. 消息生产者, 通过rpc或者http接口发送消息

消息中心模型

模型说明

  • 当message资源加载时，启动一个hub, 这个hub相当于是所有client的管理者，它负责所有client的连接和断开，如果有client要广播，也需要通过hub，hub上的属性如下:
type Hub struct {
	clients map[*Client]bool

	broadcast chan broadcastInfo

	register chan *Client

	unregister chan *Client

	registerHandler   func(*Client) error
	unregisterHandler func(*Client) error
}
  • 当有web客户端来通过websocket来连接服务器，我们会生成一个client，并且在hub上注册，这个client上包含这个connection的所有信息，也就是说和客户端的通讯要靠这个client，client属性如下
type Client struct {
	Id       string
	UserName string
	RealIp   string
	Groups   []string

	conn *websocket.Conn

	send   chan []byte
	cancel chan bool
}

消息模版说明

  • 消息模版使用go template模版语法, 示例如下:
<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <title>{{.title}}</title>
</head>
<body>
    <p>  尊敬的用户{{.username}}:</p>
 <p class="text-indent">{{.description}}, 请及时处理</p>
 </body>
</html>
  • 发送消息时候指定模版id和其中字段，如上title和username

跨服务通信支持

  • 消息中心服务支持在本服务注册的客户端的通信，即, 在目前bff和后台服务一对一的模式下，目前消息中心支持本后台服务的消息通讯。
  • 如果需要往其他服务的消息中心客户端，需要走rpc接口，发送消息
