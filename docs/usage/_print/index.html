1 - 基于BFF镜像定制产品侧BFF镜像

基于BFF镜像定制产品侧BFF镜像

BFF是通用的，它里面包含一份基础的配置文件。当产品侧使用BFF的时候，可以基于BFF基础镜像，编译属于产品的特有的BFF。

BFF镜像

BFF镜像是: docker-hub.cloud.top/cbb/bff:dev

它的工作目录是 /app ，该目录下包含执行文件bff和配置文件目录etc。

etc目录下目录结构为:

/app $: tree etc
etc
├── config-map
│   └── permission           # 权限配置文件目录
│       ├── menu.json        #前端菜单配置文件
│       └── rbac_model.conf  # casbin规则
├── i18n                     # 翻译文件
│   ├── en-US.all.json
├── rest-api.yaml            # 配置文件
├── ssl                      # https证书
│   ├── rest-api.crt
│   └── rest-api.key
└── waf                      # waf规则文件

直接使用

如上，产品侧可以直接使用BFF镜像，只需要在运行是挂载一下必要的配置文件, 一般需要挂载：rest-api.yaml menu.json ssl

这种情况下如果产品内部使用了多个BFF，配置文件挂在混了可能导致意外事情发生。

定制BFF

产品侧可以基于BFF镜像，定制属于产品自己的BFF。定制BFF只需修改里面的默认配置文件，使产品侧BFF镜像默认包含产品的配置。

Dockerfile

产品侧定制BFF时Dockerfile参考下面：

FROM docker-hub.cloud.top/cbb/bff:dev

WORKDIR /app
COPY etc /app/etc
EXPOSE 443 8080

CMD ["./bff", "-f", "/app/etc/rest-api.yaml"]

如上，主要是拷贝了etc下面的配置文件。etc下面的目录结构按上节中的创建，COPY时如果BFF已经存在对应的文件，则会覆盖原文件。

定制rest-api.yaml

下面主要列出了需要修改的字段：

# bff rpc服务名称
# 建议必须改
# 当使用服务发现时会用到这个名称作为rpc服务的名称，如果不改多个产品侧BFF服务同时存在时会出问题。
# 最好使用service名称+.rpc （service是docker-compose中的links名称，或k8s中的service名称）
rpc_name: soc.rpc 

# 服务端口
port: '9443'

# 建议必须改
certfile: ./etc/ssl/rest-api.crt
keyfile: ./etc/ssl/rest-api.key

services:
  mongodb:
    url: mongodb://mongo.topsec:27017
    # 建议必须改。
    db: bff_soc
  redis:
    url: redis.topsec:6379
  clickhouse:
    url: clickhouse.topsec:9000
    # 建议必须改。
    db: devops_logs
    username: default
    password:

middlewares:
  auth:
    enable: true
    # 建议必须改。和token签名有关系。
    secret_key: 'q4gMYQtRH9bwr73fSOXp9SVJUnYixUufY9uiedBxC8XAQqz0'

定制权限配置文件

权限配置文件在etc/config-map/permission下面，由于不同产品，菜单不一样，每个菜单调用的url也不一样，所以这下面两个文件应该是产品侧提供的。

定制BFF时，可以默认编译到定制BFF镜像里面，避免每次挂载。

定制ssl文件

各产品应该使用自己的https证书文件，不要使用默认的。

前端文件

默认BFF可以提供前端静态资源访问服务，但里面没有前端文件，需要运行时挂载。在定制BFF时，可以将前端资源文件编译进BFF。

需要将前端文件放到/srv/www/dist目录下。

定制初始化数据

每个产品的初始化数据都是不一样的，可以参考BFF中的初始化脚本写产品自身的初始化数据方法： https://git.cloud.top/cloud/BFF/-/tree/master/rest/shells/init

2 - 初始化数据

初始化脚本的使用方法

  • 可以使用init_env -h 查看初始化脚本的使用说明

  • 这个脚本接收四个参数分别是 -f -resource -dir -mode

  • -f 指定配置文件，初始化数据的数据库名，表名等都在配置文件中配置

  • -resource 指定模块，可选模块在配置中有详细列举，此为必选项，且无默认值，如果不带此参数将不会执行任何mongo数据库的初始化

  • -dir 是指定要使用的数据所在的目录（文件夹）

  • -mode 是模式，目前支持cover模式和uncover模式，cover模式将抹去以前的数据重新插入，uncover模式，如果表已经存在将会略过
